Data pipeline and storage map

How StockScreenClaude moves data into Postgres and Redis.

The app has two clocks. Scheduled and bootstrap processes hydrate data into durable Postgres tables and Redis hot state. User-facing processes then read from those stores: scans stay cache-only, APIs return persisted data, and static/MCP/assistant workflows reuse the same published artifacts.

02. What Lives in Postgres vs Redis

The shortest rule: Postgres owns durable truth; Redis owns fast coordination, Celery transport, recent cache payloads, and live operational gauges.

Postgres durable store

Data that must survive restarts and power the UI

Written by

  • Universe tasks: refresh_stock_universe, refresh_official_market_universe
  • Cache services: PriceCacheService, FundamentalsCacheService
  • Market jobs: breadth, exposure, group ranking, feature snapshot tasks
  • User work: scan creation/results, watchlists, themes, conversations, presets
  • Theme jobs: content ingestion, extraction, metrics, lifecycle policies

Read by

  • FastAPI routes: scans, stocks, breadth, groups, themes, watchlists, config
  • User scan workers: feature-run lookup, compiled criteria path, resume checkpoints
  • MCP market copilot: cited feature, breadth, group, theme, and watchlist facts
  • Static export: published runs and market artifacts
  • Governance and validation: telemetry audits, launch gates, data-quality checks
Redis hot state

Data that makes workers fast, coordinated, and observable

Written by

  • Celery clients: API routes and Beat enqueue task messages into DB0
  • Celery workers: task results and progress into DB1
  • Cache services: price, fundamentals, benchmark, FX, warmup metadata in DB2
  • Control services: data-fetch locks, market workload locks, rate limiter, circuit breaker
  • Telemetry: hot gauges and counters for operations pages and alerts

Read by

  • Workers: DB0 queues decide what process runs next
  • Operations APIs: task/result status and queue inspection
  • Scan workers: cached price/fundamental/benchmark payloads
  • Cache health and bootstrap: warmup coverage and heartbeat checks
  • Provider guardrails: throttling budgets, locks, and failure counters

03. Daily Market Pipeline

queue_daily_market_pipeline is the clearest example of the flow: one market-local chain refreshes raw data, computes market artifacts, publishes a feature snapshot, and leaves the app reading from stores instead of live providers.

1

Price refresh

smart_refresh_cache runs on data_fetch_{market}, fetches deltas, updates stock_prices, and warms Redis price keys.

2

Freshness guard

guard_price_refresh fails the chain if refresh quality drops below the minimum success threshold.

3

Breadth and exposure

Breadth reads cached prices, writes market_breadth, then exposure blends breadth with benchmark data.

4

Group rankings

IBD group jobs compute relative strength rankings and persist group history for the Groups page and MCP tools.

5

Feature snapshot

build_daily_snapshot writes feature_runs, stock_feature_daily, and a market pointer.

6

Read paths unlock

Scan creation can use the published run; dashboard, breadth, groups, static export, and assistant tools read the new state.

Manual scans are cache-only. CreateScanUseCase checks stale market data before dispatch. If no exact or compiled feature-store path can serve the request, run_bulk_scan processes chunks with batch_only_prices=True and batch_only_fundamentals=True so it does not hit upstream APIs mid-request.

04. Processes That Get Data From the Stores

These are the major reader groups. Some also write their own derived rows, but they depend on Postgres and Redis being populated first.

FastAPI and React

Live app pages

  • Scan: reads scans, scan_results, or feature-store-backed result pages.
  • Stocks: reads cached price history and fundamentals, with Postgres fallback where designed.
  • Breadth/Groups/Themes: read their durable tables and UI snapshots.
Celery user scans

On-demand computation

  • Exact match: reads a published feature_run.
  • Compile path: queries stock_feature_daily with compiled filters.
  • Async path: reads cache-only price/fundamental data and writes scan_results.
Operations

Control and monitoring

  • Task APIs: inspect Redis broker/result state and task registry metadata.
  • Cache APIs: inspect Redis health, warmup heartbeat, and Postgres cache coverage.
  • Telemetry: combines Redis hot gauges with Postgres event history.
Publishing and agents

Static and MCP surfaces

  • Static export: reads published market artifacts and cached chart/fundamental data into JSON bundles.
  • MCP market copilot: reads Postgres facts with citations.
  • Assistant gateway: persists conversations and calls the market tools.
Mental model: scheduled/background processes pay the cost of slow provider calls and durable writes. User-facing reads then stay fast: Postgres for truth, Redis for hot payloads and coordination, static bundles for read-mostly pages.
